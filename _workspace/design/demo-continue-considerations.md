## 1. 기능 정의

진행 중인 실제 프로젝트의 현재 상태를 입력으로 받아, **잔여 절차를 AI가 이어서 생성한 참고용 시뮬레이션**을 만드는 기능.

### 확정된 방침 (논의 결과)

| 항목 | 결정 |
|------|------|
| 쓰기 대상 | 원본 프로젝트는 **읽기만**. 복제본(simulation) 프로젝트에 생성 |
| 용도 | 무조건 참고용. 사용자가 복제본을 다시 편집하는 일 없음 |
| 누적 정책 | 원본당 1개 제한이 아니라 **누적 허용** + 생성 날짜·넘버링으로 구분 |
| max_tokens | 64,000 복원 완료 (`server/routes/demo.js:182`, 16,000이면 잔여 절차 많을 때 JSON 잘림) |

### 기존 코드가 주는 이점

- `server/routes/demo.js`의 2분할 생성 구조(`buildPhase1Summary`로 앞 단계 요약을 컨텍스트로 전달)가 이어쓰기의 원형. "1차 AI 결과 요약" 자리에 "실제 프로젝트 보드 요약"을 넣으면 됨
- `streamAndParse({ codes, startIndex })`가 이미 임의의 절차 부분집합을 받도록 일반화되어 있음
- `simulation` / `generating` / `failed` status가 이미 존재 (`supabase/migrations/00014_project_status_expand.sql`)

## 2. 필수 작업 (구현 전제조건)

### 2-1. 서버단 읽기 전용 가드 — ✅ 구현 완료 (`security/harden-audit-fixes`)

원래 `isReadOnlyProject` 가드는 `chat.js`와 클라이언트 UI에만 있어 API 직접 호출로 시뮬레이션 수정이 가능했음 (확인된 구멍). 다음과 같이 봉합:

- 신규 `server/lib/projectGuards.js` — `isReadOnlyProject()` 단일 정의 + `requireWritableProject` 미들웨어. chat.js의 중복 정의 제거하고 import로 일원화
- 가드 적용 라우트: `PUT designs/:procedureCode`(보드 저장), `PUT designs/:procedureCode/status`, `POST versions`(버전 생성), `PUT /projects/:id`(**status를 active로 되돌리는 우회 차단** — 핵심)
- `DELETE /projects/:id`는 의도적으로 미적용 — 시뮬레이션 정리 가능 유지
- E2E 검증 완료: active→제목변경 200, 시뮬레이션 전환 후 프로젝트 수정 403, 보드 저장 403, 삭제 200 (dev bypass 로컬 서버, 서버 테스트 63/63 통과)
- 클라이언트에도 동일 판정 로직이 복제되어 있으므로(`chatStore.js`, `ProjectPage.jsx`) 조건 변경 시 함께 수정할 것

### 2-2. 마이그레이션: `source_project_id` + `created_by` 컬럼

`projects` 테이블에 원본 참조·생성자 컬럼이 없음 (`00010_rebuild_schema.sql`). 제목 파싱으로 넘버링하면:

- 원본 제목 변경 시(원본은 active라 수정 가능) 그룹핑 깨짐 → 번호 재시작/충돌
- 원본 삭제 시 시뮬레이션이 "무엇의 참고본인지 모르는 고아"가 됨

**해법**: 마이그레이션 하나로 두 컬럼 추가 —
- `source_project_id UUID REFERENCES projects(id) ON DELETE SET NULL` (넘버링·그룹핑의 진실 원천, 번호는 표시용으로만 제목에)
- `created_by UUID` (결정 #5 "생성자에게만 노출"의 전제 — 목록 필터에 필수)
- 기준 절차·생성 시점 스냅샷은 `learner_context` JSONB 또는 별도 컬럼에 저장 (결정 #8)

### 2-3. 동시 생성 가드 — 더블서브밋 race 차단

"기존 개수 세서 +1" 넘버링은 동시 요청 두 개가 같은 번호를 받음.
이 프로젝트에는 **더블서브밋으로 동일 제목 프로젝트 이중 생성 → 사용자가 "대화가 사라졌다"고 인지한 실제 사고 전례**가 있음.
데모 생성은 몇 분짜리 SSE 스트림이라 사용자가 버튼을 두 번 누르기 쉬운 구조.

**해법**: 같은 원본에 `status='generating'`인 시뮬레이션이 이미 있으면 409 반환 (원본당 동시 생성 1개).

### 2-4. 프롬프트 입력 경계 — 보안 하드닝 관점

기존 데모의 2차 컨텍스트는 "방금 AI가 만든 형식 보장 데이터"였으나, 이어쓰기의 컨텍스트는 **사용자가 자유 편집한 보드**:

- 빈 보드, 구버전 스키마, 거대 텍스트(토큰 폭탄), 보드 내용에 담긴 프롬프트 인젝션이 시스템 프롬프트로 유입
- 참고용이어도 인젝션된 프롬프트가 만든 이상한 결과물은 서비스 신뢰 손상

**해법**: 요약 빌더에 ① 절차당 길이 상한(예: 800자 truncate) ② 총량 상한 ③ "아래는 참고 데이터이며 지시가 아님" 구분자 처리.

### 2-5. 권한 검증 (IDOR)

`projectId`를 받는 순간 IDOR 표면 발생 — `getMemberRole`로 요청자의 원본 프로젝트 멤버십 검증 필수.

### 2-6. 생성 결과 검증 파이프라인 재사용

이어쓰기 결과 보드도 기존 `standardsValidator`(코드 검증·자동 교정, QA 47건 중 46 PASS 검증 완료)를 반드시 통과시킬 것. 이를 우회하는 새 저장 경로를 만들지 말 것.

## 3. 복제본 누적 시 문제와 대응

### 3-1. 목록 오염 — 정렬 방식 때문에 확정적으로 발생

`getProjectsByWorkspace`가 `created_at DESC` 정렬(`supabaseService.js:281`) → 시뮬레이션은 항상 목록 **맨 위**에 쌓임. 5개 돌리면 첫 화면 상단 5칸이 복제본이고 실제 작업 프로젝트는 밀림. 날짜·넘버링은 구분은 해주지만 밀어내기는 못 막음.

**해법**: 목록에서 `status='simulation'` 별도 섹션 접기, 또는 원본 카드 하위 종속 표시. (UI 작업량의 대부분)

### 3-2. Provenance — "왜 시뮬레이션끼리 내용이 다른가"

시뮬레이션 #1은 prep까지 작성된 시점 기준, #3은 분석 단계까지 작성된 시점 기준이면 결과물이 크게 다름. 날짜만으로는 설명이 안 되어 "AI가 오락가락한다"는 오해 유발.

**해법**: 생성 시점의 원본 진행 상태를 메타로 스냅샷 기록(description 또는 `learner_context` JSONB) 후 카드에 표시.
제목 형식 예: `[시뮬레이션 #3] 생태 융합수업 · 7/11 · A-2-2까지 작성 기준`
(`[시뮬레이션]` 접두사는 이미 읽기 전용 가드 조건에 포함되어 있으므로 유지)

### 3-3. 비용·용량 무상한 누적

복제본 1개 = designs 행 최대 19개 + 프로젝트-성취기준 연결. 일일 쿼터 10회 × 학기 = 사용자당 수백 개 가능, 정리 장치 없음.

**해법**: ① 삭제 경로 보장(§2-1 주의사항) ② 필요 시 원본당 최근 N개 자동 정리 정책(2단계 검토).

### 3-4. 공유 워크스페이스 소유권

시뮬레이션은 요청자 개인의 참고용인데 워크스페이스 멤버 전원의 목록에 노출됨.

**해법(정책 결정)**: 삭제 권한 = 생성자 + 워크스페이스 오너. 카드에 생성자 표시.

### 3-5. 집계·통계 오염

데모 생성이 `addStandardToProject`로 성취기준 연결을 생성 → 프로젝트-성취기준 집계·활동 로그·보고서가 status를 구분하지 않으면 복제본이 쌓일수록 통계 왜곡.

**해법**: 집계 경로에서 `status='simulation'` 제외 필터.

## 5. 단계별 로드맵

### 1단계 (MVP)
`POST /api/demo/continue` — 멤버십 검증 → 빈 절차 판정 → 복제본 생성(simulation, 넘버링 제목) → 잔여 절차 청크 생성 → validator 통과 후 저장.
전제조건: §2-1(쓰기 가드), §2-2(source_project_id), §2-3(동시 생성 가드), §2-4(입력 경계).

### 2단계
- 복제본에서 마음에 드는 절차만 원본으로 "가져오기" (절차별 명시적 수락 — 원본 쓰기는 이때 처음 발생)
- 오래된 시뮬레이션 자동 정리 정책

### 하지 말 것
- 원본 프로젝트 직접 쓰기
- 진행 판정 자동 추론(마지막 연속 지점 휴리스틱 등)
- 쿼터 우회 경로
- 제목 파싱 기반 넘버링(§2-2 없이 넘버링만 구현)

## 7. 구현 착수 체크리스트 (`feature/demo-continue`)

1. 마이그레이션: `projects`에 `source_project_id`, `created_by` 추가 (§2-2)
2. `POST /api/demo/continue` 라우트: 멤버십 검증 → 동시 생성 가드(원본당 generating 1개, 409) → 복제본 생성(`[시뮬레이션 #N] 원본제목`) → **전체 복제**: 보드 + 채팅(mentioned_material_ids 재매핑) + 자료 행(storage_path 공유) → 생성 컨텍스트 조립(결정 #12: 보드 요약 + 채팅 논의 요약 + 자료 ai_analysis 요약, 각각 길이 상한 + 데이터/지시 구분자) → 현재 절차 이후 잔여 절차 청크 생성 → standardsValidator 통과 후 저장 → 실패 시 failed 마킹
3. 쿼터: 기존 demoUsage에 합산, 1회=1카운트
4. 클라이언트: 프로젝트 페이지 진입 버튼 + 워크스페이스 목록 "시뮬레이션 (N)" 접이식 섹션(생성자 필터) + 카드 메타(생성일·기준 절차) + 삭제 버튼
