### 1-A. Ds-4 "자료와 도구 연결" 전면 채우기 + 개명  ★최우선
**문제**: 앱 `Ds-2-1`(displayCode Ds-4)은 `activityFlow`·`exampleCase`가 **통째로 비어 있음**(소스에 TODO 주석 "가이드북 Ds-4 재진술 대기"). 이 절차 진입 시 교사에게 활동흐름 인트로·활동사례·협력UP 하이라이트·AI프롬프트가 **하나도 안 뜨는 실사용 결함**. 또한 최종본은 이 활동을 **"지원 도구 설계" → "자료와 도구 연결"**로 개명하고 자료(탐색/개발 구분·생성형 AI로 새 자료 제작)까지 포함하도록 재진술함.

**수정 (`server/data/procedureGuide.js` `'Ds-2-1'`)**:
- `name`: "지원 도구 설계" → **"자료와 도구 연결"**
- `coreQuestion`/`concept`/`methods`/`deliverable`: "자료+도구를 활동에 연결, 탐색/개발 구분, 공동/개별 개발, 일정·역할" 프레임으로 재진술 (생성형 AI로 기존 자료 한계를 보완·새 자료 제작 관점 포함)
- `activityFlow` 신설 (4-step, 태그 GP05→GP05→GP01→GP01):
  1. **활동별 필요 자료 나열하기** (GP05) — 각 활동에 '무엇이 있으면 학생 이해·수행이 촉진되는가' 중심으로 자료·도구를 나열, 양보다 기능 기준.
     - WITH AI 프롬프트: "다음 활동을 생동감 있게 전달하려고 함. 이를 구현하기 위한 웹 앱을 설계하여 html 파일로 제시" (생성형 AI 영상·웹앱 '생성' 기능 활용)
  2. **탐색 자료와 개발 자료 구분하기** (GP05) — 기존 활용(탐색) vs 새로 제작(개발) 구분, 생성형 AI로 '새롭게' 만들 자료 탐색. (기존/재구성/새 개발 3분류)
  3. **공동 개발과 개별 개발 구분하기** (GP01) — 루브릭·공통 안내문 등 전체 영향 자료는 공동, 차시별 활동지는 담당 개별.
  4. **(공유) 자료 개발 일정과 역할 분담 결정하기** (GP01) — 담당자·마감·중간공유·검토방법·최종수정 책임자 확정.

  1. 각 학습활동에 필요한 자료가 목적에 맞게 연결되어 있나요?
  2. 자료를 탐색·재구성·개발로 구분해 실행 계획이 분명한가요?
  3. 자료 개발의 역할·일정·공유 방식이 실제 운영 가능한가요?

## Tier 2 — 최종본 정의/의미 변경 반영

### 2-A. 성찰질문 동기화 (7개 절차: T-1~T-5, A-1, A-2)
**문제**: 이 7개 절차의 `reflectionQuestions`가 최종본 성찰질문표와 문항 단위로 어긋남(옛 일반형 유지). 나머지(A-3·A-4·Ds-1~5·DI-1·2·E-1·2)는 이미 일치.
**수정**: 각 절차 `reflectionQuestions`를 최종본 성찰질문표 문구로 교체:
- **T-1**: 궁극적 목적·핵심 가치를 하나의 명확한 문장으로 적었나 / 특정 한 사람에 치우치지 않고 모두 논의해 합의했나 (앱 3→2문항)
- **T-2**: 구체적 설계 원칙(3~5가지)을 확정했나 / 학교 교육목표·교육과정 방향과 맞닿아 있나
- **T-3**: 회의진행·기록·AI프롬프트 등 세부 과업을 빠짐없이 살폈나 / 강점·업무 여력 고려해 쏠림 없이 나눴나
- **T-4**: 갈등을 미리 다룰 실천적 규칙을 세웠나 / 팀원의 서로 다른 형편을 배려·조율했나
- **T-5**: 실행일 역산으로 단계별 핵심 목표일 포함 전체 일정을 정했나 / 누구나 언제든 확인 가능하게 공유했나
- **A-1**: 선정 전 "어떤 기준으로 고를 것인가"를 충분히 논의했나 / 확정 기준이 팀 비전·설계 방향에 맞닿아 있나
- **A-2**: 주제선정기준에 비추어 적합성 논의 후 최종 선정했나 / 선정 주제와 근거를 팀원 모두 공유·공감했나

### 2-B. A-3 "재구조화" 개념 반영 (이름은 유지)
**문제**: 최종본 세부활동 정의 = "성취기준을 분석하고 **핵심 요소를 통합하며 재구조화**한다". 앱 `A-2-1`의 `concept`/`coreQuestion`은 "세 차원 분석·연결맵" 수준에 머묾. (단, 최종본 과정표 짧은 이름은 여전히 "주제의 상세 내용 분석" → **이름 변경 불요**, activityFlow는 이미 재구조화 반영됨)
**수정**: `concept`에 "핵심 요소를 통합·재구조화하여 재구조화 성취기준을 확정한다" 취지 보강.

### 2-C. A-4 "성취수준 구체화(평가 준거화)" 반영
**문제**: 최종본 정의 = "통합 수업목표로 진술하고 **필요에 따라 평가의 준거가 될 성취수준으로 구체화**". 앱 `A-2-2`는 "통합 목표 진술"에서 끝남.
**수정**: `concept`에 "필요 시 평가 준거가 될 성취수준으로 구체화" 보강, `activityFlow` step3 설명에 "이 목표·재구조화 성취기준을 평가 준거로 구체화해 이후 평가계획과 잇는다" 문장 추가. (이름 "통합된 수업 목표 진술"은 과정표와 일치 → 유지)

### 2-D. E-2 이론 배경 반영
**문제**: 최종본 "한 걸음 더"의 3대 이론(**듀포 PLC** "학생이 무엇을 배웠나", **아지리스 단일/이중 고리** "사람 아닌 구조에서 찾기", **OECD 학습나침반 2030 AAR 사이클** 기대-실행-성찰)이 앱 `E-2-1`엔 결론("시스템 문제로 보기")만 흡수돼 있고 이론 프레임 자체는 없음.
**수정**: `notes`(또는 `concept`)에 세 이론 프레임을 간결히 반영. AI 시스템 프롬프트가 이 배경을 근거로 성찰을 유도하게 함.
