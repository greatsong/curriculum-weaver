## 2. 다음에 또 조심해야 할 것들 ⚠️

### 2-1. 인메모리 id를 Supabase FK에 넣는 사고 — **세 번째 재발이다**
- 사례: PR #4(성취기준 선택), **PR #24(데모 생성)**. `store.js`의 성취기준 id는 부팅마다 바뀌는 UUID인데
  Supabase `project_standards.standard_id` FK는 영구 id를 요구한다.
- **규칙: `addStandardToProject` 등 인메모리→DB를 가로지르는 모든 호출은 `resolveStandardId({code})`를 먼저 거친다.**
  새 기능에서 성취기준을 DB에 쓸 일이 생기면 이 규칙부터 확인할 것.

### 2-2. 조용한 실패(silent failure)가 버그를 몇 달 숨긴다
- 데모의 FK 위반은 `catch { /* 중복 무시 */ }`가 삼켜서 **프로덕션에서 성취기준 0개 연결이 무증상 지속**됐다.
- 규칙: catch로 뭉개려면 반드시 (a) 예상한 에러인지 구분하고 (b) 최소한 카운트라도 로깅한다.
  `${성공}/${전체}` 형태 로그가 있었다면 즉시 발견됐을 문제.
- 같은 계열: 로드 실패를 빈 배열/null로 폴백하지 말 것(과거 채팅 유실 버그와 동일 교훈).

### 2-3. 모듈 스코프에서 env 의존 클라이언트 생성 금지
- ESM import 호이스팅 때문에 `index.js`의 dotenv보다 **먼저** `new Anthropic(...)`이 실행된다.
  셸에 키가 없거나 빈 값이면 무키 클라이언트가 고정 → 로컬 AI 호출 전멸.
- **규칙: env를 읽는 클라이언트는 `server/lib/anthropicClient.js`처럼 lazy 싱글턴으로.** (4곳 일괄 교체 완료)

### 2-4. 기록(메모리/문서)과 코드 실상은 다를 수 있다
- 메모리에 "프로덕션 그래프가 DB에서 로드한다"고 적혀 있었지만, 실제로는 `curriculum_links`를 읽는 코드가
  한 줄도 없었다(마이그레이션 스크립트만 존재 — write-only DB). grep 한 번이면 확인되는 사실.
- 규칙: 문서·기억 기반 주장은 **grep으로 검증 후** 발화. 특히 "~가 연결돼 있다"는 배선 주장.

### 2-5. 일회성 자동 동작 vs 사용자 조작의 레이스
- 검색 줌아웃 버그의 정체: `onEngineStop`의 최초 `zoomToFit`이 수 초~수십 초 뒤에 발동해
  사용자의 검색 줌인을 덮어씀. **비결정적 버그는 "타이밍을 바꿔가며" 재현**해야 잡힌다(로드 직후 조작 등).
- 규칙: 지연 실행되는 자동 카메라/스크롤/포커스는 사용자가 먼저 조작하면 취소한다(`initialFitDoneRef` 패턴).

### 2-6. 전역 임계값은 "반드시 포함돼야 할 것"을 놓친다
- 데이터 과학↔인공지능 기초는 명백히 밀접한 과목쌍인데 코사인이 0.55~0.63이라 전역 컷(0.6)으로는 2개만 통과.
  상호보완적 내용은 어휘 유사도가 중간대에 머문다.
- 규칙: 선발 로직 설계 시 "무엇이 반드시 포함돼야 하는가"를 먼저 정의하고,
  전역 컷 대신 **단위별 보장형 선발**(과목쌍별 top-N 등)을 검토한다.

### 2-7. 개수 목표는 품질을 희생시킨다
- v1 링크 생성의 통과 기준이 "총 2,000개 이상" 같은 개수 지표뿐이라 약한 링크가 대량 유입
  → 재판정에서 26% 기각, 42%가 게시 기준(0.7) 미달.
- 규칙: 생성 파이프라인에는 반드시 품질 게이트(LLM 판정 점수, 표본 검수)를 함께 설계한다.
  게시 정책은 한 문장으로 설명 가능해야 한다("게시 링크는 전부 품질 0.7 이상").

### 2-8. 장기 배치 실행은 로그·진행상태·복구 경로가 생명
- 같은군 생성 1차 실행에서 네트워크 장애로 223배치가 실패했지만,
  `| tail` 파이프로 로그를 흘려버려 실패 원인 파악이 늦었다. 대신 progress.jsonl(파라미터 포함 batchId),
  결과 jsonl, `--import-results` 복구 경로 덕분에 **데이터 유실 0**으로 재개할 수 있었다.
- 규칙: 장기 실행은 ① 로그 전체를 파일로 ② 진행상태는 파라미터가 각인된 키로 ③ 결과는 즉시 append
  ④ DB 적재 실패 시 파일에서 복구하는 경로를 처음부터 만든다.

### 2-9. 로컬에서 인증 필요한 플로우는 "테스트 계정 + JWT" 기법으로 검증
- dev 더미 클라이언트 ↔ 실 JWT 서버 불일치 때문에 워크스페이스/데모 플로우는 로컬 브라우저로 E2E 불가.
- 해결 기법(재사용 가능): admin API로 테스트 유저 생성 → `signInWithPassword`로 JWT 획득 →
  curl로 API 직접 호출(SSE 포함) → DB로 결과 검증 → **테스트 데이터 완전 삭제**.
  이 기법으로 데모 버그 2건을 수정 전/후 정량 비교(0/19 → 19/19)까지 검증했다.

### 2-10. 가이드는 문서가 아니라 제품 안에
- "가이드 페이지를 만들까?"의 답은 예시 버튼(원클릭 체험) > 첫 방문 코치마크 > 기존 가이드 확장 순이었다.
  가이드 자산이 이미 4개나 병존하고 있었다는 사실 자체가 경고 — 새 문서 추가 전에 기존 자산 인벤토리부터.

## 3. 잘 작동한 프로세스

- **재현 우선**: 줌아웃 버그를 코드 추측이 아니라 브라우저 타이밍 재현으로 확정 → 한 줄 수정으로 종결.
- **dry-run 계층**: 모든 파이프라인에 비용 0의 dry-run → 소액 스모크(--limit) → 전체 실행 순서를 강제.
  임계값·규모·비용을 실측으로 결정할 수 있었다.
- **비파괴 원칙**: 강등은 삭제가 아닌 candidate 보관, 하이드레이션 실패 시 정적 파일 폴백,
  upsert는 ignoreDuplicates — 되돌릴 수 있는 상태를 유지하며 전진.
- **점검 요청을 E2E로**: "잘 작동하는지 점검"에 코드 리딩만 했다면 두 버그 모두 못 잡았다.
