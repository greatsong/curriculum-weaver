## 0. 전제 — 현재 구조 요약

- 워크플로우: **6 Phase (prep/T/A/Ds/DI/E) × 19 절차(prep + 18) × 128 스텝**
- 내부 코드(`T-1-1`, `A-2-1` 등)는 DB·API 전용. 사용자에게는 **displayCode**(`T-1`, `A-3` 등)만 노출
  - 주의: 내부 코드와 표시 코드가 **어긋나 있다** (T-1-2→T-2, T-2-1→T-3, A-1-2→A-2 등). "틀은 같다"는 말은 displayCode 체계(T-1~T-5, A-1~A-4, Ds-1~Ds-5, DI-1~2, E-1~2) 유지를 뜻하는 것으로 해석
- DB(designs·messages·projects)는 **절차 코드만 저장**하고 문구는 저장하지 않음 → 문구만 바뀌면 DB 마이그레이션 불필요

## 1. 변경 시나리오 분기 — 새 자료를 받으면 가장 먼저 판정할 것

새 안내 자료가 도착하면 아래 4개 레벨 중 어디까지 해당하는지부터 판정한다. 레벨이 올라갈수록 작업량이 계단식으로 늘어난다.

| 레벨 | 바뀌는 것 | 추가 작업 |
|------|----------|----------|
| **A. 설명·안내 문구만** | description, 가이드 본문, 스텝 설명 | 정의 파일 4개 교체로 끝. DB 무관 |
| **B. 절차 이름(name)도** | + PROCEDURES의 name | + 하드코딩 5건 + 테스트 1건 + 가이드북 재생성 |
| **C. 스텝 구성·보드 구조도** | 스텝 개수/순서, boardField, 보드 스키마 필드 | + localStorage 스텝 커서·messages.step_context 어긋남 처리, 기존 프로젝트 보드 JSONB 호환 검토 (§6.2) |
| **D. 원리·정합성 체인도** | 협력 원리 57개, checkAgainst 관계 | + principles.js 갱신, coherenceCheck 재설계 (§6.3) |

레벨 C·D가 포함되면 `procedure_simplification_upcoming.md` 메모리의 구조 변경 조사(SSOT 6군집, 코어 5절차, PR 순서)를 함께 읽을 것.

## 2. 문구의 정본 — 정의 파일 4개 (여기만 고치면 대부분 자동 반영)

### 2.1 `shared/constants.js` (~850줄)
- `PHASES` — 6개 Phase의 name·icon·color
- `PROCEDURES` — 19개 절차의 **name·description·displayCode** (내부 코드가 키)
- 파생: `PROCEDURE_LIST`(order 정렬), `UNSKIPPABLE_PROCEDURES`(코어 5), 스킵 관문 함수들, `getProcedureDisplayCode`, `replaceInternalProcedureCodes`(AI가 내부 코드를 노출해도 표시 코드로 치환하는 결정적 가드 — 유지 필수)

### 2.2 `shared/procedureSteps.js` (~1,400줄)
- `PROCEDURE_STEPS` — 절차별 5~9개 스텝, 총 128개. 각 스텝: `title`·`description`(문구) + `actionType`·`actorColumn`·`aiCapability`·`boardField`(구조 메타)
- **문구 교체는 title·description만. 구조 메타를 건드리면 레벨 C가 된다** (boardField는 보드 스키마 필드명과 결합, aiCapability는 AI 프롬프트의 스텝별 역할 결정)

### 2.3 `server/data/procedureGuide.js` (~1,300줄) — **가장 큰 덩어리이자 AI 행동의 원천**
- `PROCEDURE_GUIDE` — 절차별: `coreQuestion`(핵심질문), `concept`(개념), `methods[]`(방법), `deliverable`(산출물), `aiRole{guide,generate,check,record}`(AI 역할), `coherenceCheck{checkAgainst[],description}`(정합성 점검 대상), `notes`(유의사항), `reflectionQuestions[]`(성찰질문), `activityFlow[]`(활동흐름 — collaborationTag·aiPrompt 예시 포함), `exampleCase`(사례)
- 소비처: ① `aiAgent.js` 시스템 프롬프트에 동적 주입(AI의 절차별 대화 프로토콜), ② `chat.js`의 `POST /procedure-intro`(절차 소개 메시지를 이 데이터로 결정적으로 조립 — AI 호출 아님)
- `getCoherenceTargets`/`getAllCoherenceChecks`가 checkAgainst를 읽음. **checkAgainst 배열은 문구가 아니라 절차 간 관계** — 안내 내용이 바뀌면 이 관계도 유효한지 재검토 (현재 6개 절차에 정의: T-1-2→[T-1-1], A-1-2→[T-1-1,A-1-1], A-2-2→[A-2-1], Ds-1-1→[A-2-2], Ds-1-3→[Ds-1-1,A-2-2], Ds-2-2 인근→[A-2-2,Ds-1-1])

### 2.4 `shared/boardSchemas.js` (~510줄)
- `BOARD_SCHEMAS` — 절차별 보드(산출물 JSONB)의 필드 정의: `label`·`description`(문구) + `name`·`type`·`required`·`options`(구조)
- **label·description만 바꾸면 레벨 A. name(필드 키)을 바꾸면 기존 프로젝트의 designs.content JSONB와 어긋난다 = 레벨 C**

### 2.5 (레벨 D일 때) `server/data/principles.js` + `generalPrinciples.js`
- 절차별 협력 원리 57개(`substep` 필드로 절차 코드에 결합). **원문 "유의사항"에서 추출된 것이라 세부 안내가 바뀌면 원리도 함께 개정될 가능성 높음** — 새 자료에 원리 변경이 포함되는지 반드시 확인
- 서빙 경로: DB 아님. `store.js`가 파일에서 인메모리 시드(구 DB 테이블은 00010에서 DROP됨). 관리자 PUT은 인메모리만 수정(재시작 시 파일값으로 복귀) — **영구 변경은 파일 수정이 정답**

## 6. 함정 목록 — 새 세션이 놓치기 쉬운 것

### 6.1 절차 인트로는 프로젝트별로 1회 생성 후 메시지로 영구 저장된다
- `chatStore.js`의 `introCache`는 **DB에 저장된 메시지에서 파생**(loadMessages가 절차별 인트로 추출). 이미 방문한 절차는 재생성하지 않음(`introCache[code]` 존재 시 스킵)
- → 문구 교체 후에도 **진행 중인 기존 프로젝트는 옛 안내 인트로를 계속 본다.** 이게 허용되는지(역사적 기록으로 수용 vs 재생성 제공) **사용자 정책 결정 필요**. 재생성하려면 저장된 인트로 메시지 처리 방안 필요 — 기본 권장은 "기존 프로젝트는 그대로, 신규부터 새 안내"

### 6.2 (레벨 C일 때) 스텝 위치의 이중 저장
- 클라 `procedureStore.stepMemory` → `localStorage['cw_steps_{projectId}']`에 절차별 스텝 번호 저장. 스텝 개수가 줄면 저장된 번호가 범위를 벗어남 → 로드 시 클램프 가드 필요
- DB `messages.step_context INTEGER` — 과거 메시지의 스텝 참조는 역사 기록으로 방치 가능
- 보드 JSONB: `designs.content`가 `boardSchemas`의 필드 키(name) 기준. 필드 키 변경 시 기존 데이터 마이그레이션 또는 리더 호환 필요. `UNIQUE(project_id, procedure_code)` 제약 유의

### 6.3 (레벨 D일 때) 원리·정합성은 문구가 아니라 "관계"다
- checkAgainst(어느 절차가 어느 절차와 정합성을 점검하나)와 원리의 substep 결합은 새 안내의 교육 설계 논리에 맞게 **다시 설계**해야 하는 것이지 문구 치환 대상이 아님. 새 자료에 이 관계 정의가 없으면 사용자에게 질의

### 6.4 demo.js의 이중 SSOT
- `demo.js:30-31` `PHASE1_CODES`/`PHASE2_CODES`가 절차 코드 목록을 **별도 하드코딩** (PROCEDURE_LIST 파생 아님). 코드 집합이 유지되면 무해하나, 레벨 C에서 절차가 늘거나 줄면 반드시 동기화. 데모 프롬프트에는 절차 이름이 `procedureNameMap`으로 들어가므로 이름 변경은 자동 반영

### 6.5 chat.js 파서의 절차 코드 미검증
- AI XML 응답의 procedure 코드를 chat.js는 검증 없이 통과(designs.js는 검증). 문구 교체로 AI가 새 안내를 학습하는 과정에서 이상 코드를 뱉으면 조용히 이상 데이터 생성 가능 — 교체 작업과 무관하게 검증 추가 권장 (기존 조사에서 이미 지적된 사항)

### 6.6 무관한 것 (건드리지 말 것)
- 성취기준 데이터(`standards.js` 4,856개)·임베딩 캐시·curriculum_links — 절차 문구와 완전 무관
- `server/data/stageGuide.js` — main에는 없음(워크트리 잔재). `DesignBoard.jsx`·`StageNav.jsx` — 미사용 레거시
- `.claude/worktrees/` 아래 파일들 — grep 시 제외할 것 (구버전 사본이라 오염됨)

## 7. 권장 작업 순서

1. **새 자료 수령 + 레벨 판정** (§1). 형식은 "절차 코드(내부+표시) 기준 대조표" — 현행 문구 인벤토리를 4개 정의 파일에서 추출해 옆 칸에 새 문구를 받는 방식이 누락 방지에 최선
2. **베이스라인**: 현행 테스트 통과 확인 + dev 서버로 절차 소개·보드·보고서·데모 스모크 (포트 4006/4007, 서버 `.env`의 `DEV_AUTH_BYPASS=true` 활용 가능)
3. **정의 파일 4개(+원리) 교체** — 한 브랜치, 절차 단위 커밋 권장. 4개 파일은 서로 결합돼 있어 **같은 절차는 같은 커밋**에서 함께 수정
4. **하드코딩 5건 + 테스트 기대값** 갱신, 새 절차명으로 grep 재감사
5. **검증**: vitest + dev 서버에서 신규 프로젝트 생성 → 절차 소개(procedure-intro) 새 문구 확인 → AI 대화 1~2절차 → 보고서 생성 → 데모 시뮬레이션 → 스킵 UI(취소선·분모) 정상
6. **가이드북 재생성**: 스크린샷 재촬영 → generate-guidebooks.cjs 카피 수정 → docx/PDF 4종 재생성
7. **기존 프로젝트 정책 확정** (§6.1 인트로) 및 필요 시 안내 배너 등 후속

## 8. 사용자에게 확인할 질문 (착수 시)

1. 절차 이름(name)도 바뀌나요, 설명·안내만인가요? (레벨 A vs B)
2. 스텝 구성이나 보드 산출물 구조도 바뀌나요? (레벨 C 여부)
3. 협력 원리 57개와 정합성 점검 관계도 개정되나요? (레벨 D 여부)
4. 진행 중인 기존 프로젝트는 옛 안내를 유지해도 되나요? (§6.1)
5. 새 문구 자료는 어떤 형식으로 주실 건가요? (대조표 제공 가능)
